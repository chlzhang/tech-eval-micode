# 报告结构参考

> 章节内的小标题可根据技术主题弹性调整，但主干顺序与编号不可更改。默认生成精简版，加 `--full` 参数生成完整版。

## 精简版报告结构（4章，默认）

### 一、执行摘要

与完整版相同：开篇即给结论，三到五句话讲清总体判断、关键理由、潜在价值与最大障碍。

### 二、交流背景

精简为 2-3 段：对方是谁、交流形式与时间、核心技术主题。去掉冗余的公司介绍细节。

### 三、对方技术主张（事实层）

> 本章所有内容均为"对方主张"，每条必须标注来源（口头 / 对方文档）。

保留核心指标表格，去掉与初步结论重复的文字描述。分四块：
1. **核心技术原理** — 一句话概括技术路线
2. **关键性能指标** — 表格呈现，保留具体数值
3. **技术成熟度阶段** — 简述当前阶段
4. **已有应用案例** — 简述规模

### 四、初步结论（合并完整版第五～八章）

> **本章必须避免内容重复。** 每个冲突点、风险点、结论只在一个位置完整论述一次，其他位置如需引用则用一句话交叉引用。

**4.1 核心发现（不超过5条）**
- 从行业基准中提取最关键的冲突点，从多维评估中提取最核心的判断
- 每条结论附一条依据，不展开论述
- 标注判断类型（基于充分信息 / 信息不足下的初步判断）

**4.2 量化分析**
- 用文字摘要呈现核心冲突点和关键经济数据结论（如年毛利区间、投资回收期），不展开表格
- 行业基准对比表、投资与收益替代估算表作为附表放在报告末尾（见附表定义），正文用"详见附表N"交叉引用
- 基于行业基准做替代估算，正文仅保留结论，详细计算过程放附表

**4.3 推荐方案与决策路径**
- 单一推荐方案（非多选）
- 投资框架用一句话概括我方核心投入，详细投入对比表作为附表放在报告末尾（见附表定义），正文用"详见附表N"交叉引用
- 决策节点 if-then 表（保留在正文）
- 退出条件清单
- 关键时间线节点
- 责任分工建议

**4.4 主要风险**
- 只保留严重程度为"高"的风险，中低风险移入待核实清单
- 每条风险一句话描述，不展开

**4.5 待核实清单**
- 保留全部待核实问题（这是下次交流的提问清单）
- 按优先级排列，标注建议核实方式

### 附表（精简版和完整版通用）

支撑性数据表格从正文剥离，集中放在报告末尾（附注之前），让正文保持精炼。附表按编号引用，正文对应位置改为"详见附表N"。

**默认附表（按需增减）：**
- **附表1：行业基准对比**（对应正文4.2/第五章）— 对方主张 vs 行业基准的逐项对比表
- **附表2：投资与收益替代估算**（对应正文4.2/第六章）— 详细测算过程，含假设、来源和计算
- **附表3：投资框架——我方与对方投入对比**（对应正文4.3/第七章）— 投入分项对比

**附表规则：**
- 附表编号从1开始连续编号，正文引用格式为"详见附表N"
- 每张附表标题格式为"附表N：标题（对应正文X.X节）"
- 附表中的数据必须与正文摘要一致，不得出现正文未提及的数据
- 如某张附表数据不足（如无行业基准），可省略该附表并在正文中说明

## 完整版报告结构（8章，--full 参数）

以下为固定主干章节。

## 一、执行摘要

开篇即给结论，三到五句话讲清：
- 这是什么技术
- 总体判断（值得深入探索 / 谨慎观望 / 不建议跟进）
- 最关键的一两条理由
- 对我方业务升级转型的潜在价值与最大障碍

## 二、交流背景

简要交代：
- 对方是谁（单位、主营业务）
- 交流/考察的形式（线上会议 / 实地考察 / 展会交流）与时间
- 涉及的核心技术主题

## 三、对方技术主张（事实层）

> 本章所有内容均为"对方主张"，每条必须标注来源（口头 / 对方文档）。

分四块呈现：
1. **核心技术原理** — 对方描述的技术路线与底层逻辑
2. **关键性能指标** — 尽量保留具体数值，不四舍五入
3. **技术成熟度阶段** — 实验室 / 中试 / 量产
4. **已有应用案例与客户** — 规模、场景、运行时长

## 四、我方现场观感（仅当素材含我方实地观察时）

分两层：
1. **客观观察事实** — 我方看到了什么（产线状态、设备实物、运行数据等）
2. **由此产生的判断** — 基于观察的分析，必须与第二章的"对方口头主张"严格区分

## 五、行业基准对照

> **筛选原则：** 只纳入与"我方要不要参与、怎么参与"这个核心决策直接相关的基准。与决策无关的行业通用数据（如环保排放指标、预处理工艺参数等）不纳入正文，仅在对方主张与之冲突时才引用。

- 该技术领域与我方决策相关的核心指标、标杆值、已知瓶颈
- 每条标注来源（我方沉淀文档 / 现场检索 / 模型知识）
- **显式列出**对方主张与基准之间的差距或冲突（铁律三）
- **行业基准的决策含义：** 每条基准不仅列出数据，还需说明"这条基准对我方决策意味着什么"

## 六、多维评估（判断层）

每条结论必须附依据，区分"基于充分信息的判断"与"信息不足下的初步判断"。

1. **先进性** — 相对行业标杆，这项技术领先、持平还是落后；领先在哪、含金量多高
2. **可行性** — 拆三个子维度：
   - 技术可行性：原理是否成立、有无反例
   - 工程可行性：能否规模化、良率如何
   - 经济可行性：成本对比、投资回收周期
3. **量化分析（必须项）：**
   - 当对方未提供经济数据时，**必须使用行业基准数据做替代估算**，给出合理区间而非留白
   - 正文用文字摘要呈现核心数据结论，详细测算过程（含假设、来源和计算）作为附表放在报告末尾，正文用"详见附表N"交叉引用
   - 单位不同的数据必须先换算为统一单位再比较（如月产能→年产能）
   - 估算结果标注"基于行业数据替代估算，待对方提供实际数据验证"

## 七、对我方的适配性与决策路径

> **本章不是任务清单，而是决策路径。** 必须回答"我方下一步具体做什么、什么条件下推进、什么条件下退出"。

- 与公司业务升级转型的契合点在哪
- 引入需要哪些配套条件、门槛高低
- **决策路径（必须包含以下要素）：**
  1. **推荐方案：** 明确推荐的合作模式和参与深度，而非列出多种选项让用户自选
  2. **投资框架：** 用文字概括我方核心投入和投资量级，详细投入对比表作为附表放在报告末尾，正文用"详见附表N"交叉引用
  3. **决策节点：** 用 if-then 结构描述关键决策条件（如"若对方在 X 个月内提供了 Y 数据且验证通过，则推进到 Z 阶段"）
  4. **退出条件：** 明确什么情况下应终止探索（如"若关键技术指标经验证不达标、或政策环境发生不利变化、或经济测算不支持"）
  5. **时间线：** 给出建议的行动时间表（含具体月份或周数）
  6. **责任分工：** 建议由哪个部门/岗位牵头推进

## 八、风险与待核实清单

汇总两类内容：
1. **识别出的主要风险** — 技术风险、市场风险、合作风险等
2. **待核实问题清单** — 铁律二中积累的所有"对方未答清、素材未覆盖"的问题，逐条列出，作为下次交流或尽调的提问清单
